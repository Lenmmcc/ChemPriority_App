# 一键批量查询细粒度进度设计

## 目标

让一键批量查询在网络模块耗时较长时，清楚显示整体执行位置、当前模块已提交/已完成数量、正在等待的化合物、最近一条请求耗时和该模块的单条超时上限。

## 问题与原因

当前 `run_ordered_batch()` 只在单条任务结束时调用进度回调；页面将该回调直接映射到唯一进度条。因此 EPI Suite 等待单条远端响应时，界面没有新状态，且不同模块会从 0 重新开始，用户难以判断它是在等待、重试还是卡住。

## 方案

### 批处理事件

在保留现有 `progress_callback(done, total, label)` 兼容性的前提下，为 `run_ordered_batch()` 增加可选事件回调。事件包含：

- `started`：主线程提交一条任务时发出，带任务标签、序号、总数和开始时间。
- `completed`：任务正常结束时发出，带完成数、标签和本条耗时。
- `failed`：任务以异常结束时发出，带完成数、标签、耗时和错误摘要。

并发批处理会在初始任务和每个补位任务提交时发出 `started`；完成事件仍按输入顺序汇总结果，不改变现有结果顺序或错误隔离方式。

### 工作流转发

一键工作流将批处理事件映射为带模块名称的活动事件，应用于标识符补全、EPI、CompTox、ECHA 用途、ECHA GHS 和来源属性。既有完成进度回调保留，供其他页面继续使用。

### 页面显示

一键页面在执行期间显示：

1. 总进度条：已完成模块数 / 已选模块数；本地筛查、标识符、EPI、CompTox、ECHA 用途、ECHA GHS、来源属性和可选 Pov-LRTP/ToxPi 各占一个阶段。
2. 当前模块进度条：已完成条数 / 总条数。
3. 活动状态：已提交数、处理中数、已完成数、最近开始的化合物、最近结束耗时；并列出最多三个正在处理的化合物。
4. EPI 等模块的等待提示：显示该模块单条请求超时上限（例如 EPI 默认 90 秒），而不显示会失真的倒计时。

页面仅根据回调事件更新，不新增网络请求、后台线程或前端轮询；因此当远端在单条请求期间完全无响应时，状态会保持为“正在等待”，这是准确而非静止的反馈。

## 兼容性与边界

- 不改变网络请求参数、超时、缓存、并发度、查询顺序或导出内容。
- 已使用旧 `progress_callback` 的页面和函数无需修改。
- 事件回调为可选；未提供时批处理行为与当前版本一致。
- 只调整一键页面的展示，不改变第三页或第四页的页面布局。

## 验证

- 为顺序与并发批处理增加 started/completed/failed 事件测试，同时保留旧完成回调测试。
- 为自动工作流增加模块活动事件转发测试。
- 为一键页面增加进度组件和等待提示的页面契约测试。
- 运行完整单元测试与 `compileall`。
