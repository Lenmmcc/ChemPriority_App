# 一键批量查询模块级检查点与部分结果下载设计

## 目标

修复“一键批量查询”必须等待全部模块、全部图表和最终 ZIP 生成完毕后，页面才保存并展示任何结果的问题。新流程在每个模块完成、失败或跳过后立即建立检查点，使已经产生的结果可以查看、下载，并在页面刷新或短暂断线后恢复。

## 问题与原因

当前页面先同步执行 `run_auto_query_workflow()`，随后生成全部图表和最终 ZIP，最后才把结果写入 `st.session_state`。对于包含数百个唯一化合物的输入，多项联网模块可能持续运行几十分钟或更久。如果浏览器刷新、连接中断、服务器进程退出，或最终图表与 ZIP 生成失败，本次运行此前已经完成的模块没有独立的页面结果和下载入口。

查询缓存可以减少再次联网的成本，但查询缓存不是用户结果检查点：它不记录哪些模块已经完成，不包含页面结果表、运行日志、图表和可直接下载的模块工作簿，也不能保证中断后立刻向用户交付已完成结果。

## 设计原则

- 以模块边界建立检查点，不在每个化合物完成时频繁写盘。
- 保留现有同步工作流、查询顺序、缓存键、超时、并发度和结果表协议。
- 检查点能力必须可选；未提供回调的现有调用者行为不变。
- 检查点写入失败不得中止化学查询；应退化为当前会话内保存并记录警告。
- 单个模块失败或最终导出失败不得删除此前成功模块的结果。
- 运行期间只生成新完成模块的独立工作簿，不重复生成累计 ZIP。
- 不建设永久任务历史、用户账户、后台任务队列或跨服务器对象存储。
- 第一版恢复已完成结果，但不自动从下一个模块续跑；用户重新执行时复用现有查询缓存。

## 范围

本设计只调整“一键批量查询”页面和其共享工作流：

- 模块级检查点事件；
- 当前会话即时结果；
- 24 小时服务器临时检查点；
- 已完成模块的即时 Excel 下载；
- 失败或恢复状态下的部分结果 ZIP；
- 页面刷新后的结果恢复；
- 过期清理、显式清理和安全边界；
- 对应的单元测试、页面契约测试和合成长任务验证。

本设计不改变第三页、第四页等独立查询页面的布局，不改变任何外部接口请求，不增加自动重试，不改变现有完整 ZIP 的目录和工作簿协议。

## 方案选择

### 采用：可选检查点回调

为 `run_auto_query_workflow()` 增加可选 `checkpoint_callback`。工作流仍然同步运行并最终返回 `AutoWorkflowResult`；每个模块达到终态后，额外向回调发送当前累计结果的只读快照。

该方案复用现有 `record()`、`run_step()`、`AutoWorkflowResult.tables` 和模块导出映射，改动范围小，且不会让页面复制工作流编排逻辑。

### 不采用：生成器逐段返回

将工作流改写为生成器可以自然地 `yield` 模块结果，但会改变公开返回协议，影响大量现有测试和调用者，迁移成本高于本次需求。

### 不采用：页面直接编排模块

让页面逐个调用标识符、EPI、CompTox、ECHA 等函数可以快速显示结果，但会形成第二套依赖顺序和错误处理逻辑，后续容易与 `src/auto_query_workflow.py` 漂移。

## 检查点事件协议

新增不可变数据对象 `AutoWorkflowCheckpoint`，包含以下字段：

- `run_id: str`：本次运行的随机标识；
- `input_signature: str`：上传文件字节的 SHA-256；
- `settings_signature: str`：当前列映射、模块选择和运行设置的稳定签名；
- `selected_steps: tuple[str, ...]`：本次选择的模块顺序；
- `finished_steps: tuple[str, ...]`：已经达到完成、失败或跳过终态的模块；
- `current_step: str | None`：刚刚达到终态的模块；
- `status: str`：`running`、`completed` 或 `failed`；
- `result: AutoWorkflowResult`：当前累计表、日志、警告和已有图表；
- `error_message: str`：顶层失败摘要；正常运行为空；
- `updated_at: str`：UTC ISO 8601 时间。

`checkpoint_callback(checkpoint)` 在工作流协调线程上同步调用，不新增后台线程。工作流通过统一的 `emit_checkpoint()` 包装调用；若回调抛出异常，`emit_checkpoint()` 将错误写入 `warnings` 的 `Checkpoint` 阶段并继续后续模块，不让检查点故障中止化学查询。回调触发点为：

1. 本地筛查完成或失败；
2. 标识符补全完成或失败；
3. EPI 完成、失败或因缺少 SMILES 跳过；
4. CompTox 完成或失败；
5. ECHA REACH 用途完成或失败；
6. ECHA GHS/C&L 完成或失败；
7. 来源属性完成或失败；
8. Pov-LRTP / PBM / ToxPi 完成或失败；
9. 工作流正常结束，发送 `completed` 检查点；
10. 页面捕获未处理异常后，补写 `failed` 检查点。

单条化合物的 `started`、`completed`、`failed` 活动事件仍只服务进度显示，不触发检查点写盘。

## 检查点存储

新增独立模块 `src/auto_query_checkpoint.py`，负责创建、保存、加载、删除和清理检查点。默认根目录为：

```text
.cache/auto_query_runs/
```

页面生成高熵随机恢复令牌，并把令牌放入查询参数 `run`。服务器目录使用恢复令牌的 SHA-256 作为名称，避免直接把令牌或化合物信息写入路径：

```text
.cache/auto_query_runs/<token_sha256>/
├── manifest.json
├── tables/
│   ├── Structure_Preparation.json.gz
│   ├── Identifier_Completion.json.gz
│   ├── EPI_Results.json.gz
│   └── CompTox_Summary.json.gz
├── charts/
│   ├── <chart_key>.png
│   └── <chart_key>.pdf
└── modules/
    ├── local_screening.xlsx
    ├── identifier_completion.xlsx
    ├── epi_suite.xlsx
    └── comptox_use.xlsx
```

### 清单文件

`manifest.json` 使用版本化结构，至少包含：

- `schema_version: 1`；
- 令牌哈希；
- 输入文件名和输入签名；
- 设置签名；
- 选择模块和已完成模块；
- 当前状态与刚完成模块；
- 运行日志和警告摘要；
- 表名、图表键和模块工作簿文件清单；
- 创建时间、更新时间和过期时间；
- 顶层错误摘要。

清单不保存 API 密钥、Cookies、访问令牌或服务器绝对路径。

### 表格格式

每个 DataFrame 使用 pandas `orient="table"` JSON 并通过 gzip 压缩，保留列名、基础类型和缺失值语义。检查点不使用 Pickle，避免加载磁盘文件时引入任意代码执行面。

每张表和每个清单文件都先写到同目录临时文件，刷新并关闭后使用 `os.replace()` 原子替换正式文件。加载时以 `manifest.json` 为提交标记；清单不存在或不完整的目录视为损坏检查点。

### 模块工作簿

模块达到终态后，只生成该模块的独立 XLSX，并复用 `AUTO_WORKFLOW_EXPORT_MODULES`、`PUBLIC_TABLE_NAMES` 和现有模块工作簿生成逻辑。已生成的模块工作簿不在后续模块完成时重复创建。

正常运行结束时仍只生成一次现有完整 ZIP。若运行顶层失败，或页面从临时检查点恢复，则可将已有模块 XLSX、运行日志和警告打包成“部分结果 ZIP”；部分 ZIP 不伪装成完整结果，文件名和页面文字必须明确包含 `Partial` / “部分结果”。

## 页面数据流

### 新运行

1. 用户上传文件并确认设置。
2. 页面计算输入签名和设置签名。
3. 用户点击“开始一键运行”后创建恢复令牌和 `run_id`，写入 `st.query_params["run"]`。
4. 页面初始化运行清单，并创建“已完成模块，可立即下载”容器。
5. 页面调用 `run_auto_query_workflow(prepared_input_df, config=config, progress_callback=update_progress, activity_callback=update_activity, checkpoint_callback=save_checkpoint)`。
6. 每次回调依次完成：
   - 更新 `st.session_state` 中的累计结果；
   - 生成刚完成模块的 XLSX；
   - 原子保存服务器临时检查点；
   - 更新已完成模块状态和模块下载按钮。
7. 正常结束后生成图表和完整 ZIP，把清单状态更新为 `completed`。

### 页面恢复

1. 页面加载时读取查询参数 `run`。
2. 严格校验令牌字符集和长度，计算目录哈希，并确认解析后的路径仍位于检查点根目录内。
3. 加载未过期的 `manifest.json`、表格、图表和模块工作簿。
4. 页面显示“已恢复上次运行的部分结果”，恢复结果总览、运行日志和下载入口。
5. 如果恢复的是 `running` 状态，将其显示为“上次运行未正常结束”，不声称后台仍在执行。
6. 第一版不自动继续未完成模块；用户可以重新点击运行，现有网络查询缓存会减少重复请求。

若用户随后上传不同输入，页面不得把新输入与旧检查点合并。开始新任务时清除当前页面的旧 `run` 参数并创建新令牌；旧检查点等待过期清理，或由“清空当前数据”显式删除。

## 页面交互

运行期间新增“已完成模块，可立即下载”区域。每个模块达到终态后显示：

- 模块名称和完成、失败或跳过状态；
- 结果行数和警告数量；
- 模块独立 Excel 下载按钮；
- 失败模块的简短错误摘要；
- 已有关键结果表的预览。

下载按钮使用 `on_click="ignore"`，避免点击下载触发页面重跑并中断仍在执行的工作流。该能力从 Streamlit 1.43.0 起提供，因此 `requirements.txt` 的最低版本调整为 `streamlit>=1.43,<2`。

恢复页面和失败页面额外显示“下载部分结果 ZIP”。正常完成后继续显示现有完整 ZIP 下载按钮，且完整 ZIP 的文件名、目录结构和表格协议不变。

## 会话状态

现有结果键继续保留，并新增：

- `auto_query_run_token`；
- `auto_query_checkpoint_manifest`；
- `auto_query_partial_result`；
- `auto_query_module_workbooks`；
- `auto_query_checkpoint_warning`。

输入签名或设置签名改变时，当前会话结果仍按现有规则失效。磁盘检查点只有在开始新运行、显式清理或过期清理时删除，避免普通控件重跑误删可恢复结果。

## 失败与降级规则

- 单个化合物失败：沿用现有批处理警告表；模块完成后正常保存检查点。
- 整个模块抛出异常：沿用 `run_step()` 隔离，记录失败模块，并保存此前累计结果。
- 检查点表格或清单写入失败：查询继续，页面显示“临时恢复保存失败，本次结果仅保留在当前页面会话”；后续模块仍可再次尝试保存。
- 模块 XLSX 生成失败：保存其他表格和清单，模块状态保持完成，但该模块下载按钮显示导出失败。
- 图表生成失败：不影响表格检查点和模块 XLSX。
- 最终完整 ZIP 生成失败：把运行标记为 `failed`，保留模块下载和部分结果 ZIP。
- 恢复文件损坏：隔离该检查点，显示可理解的警告，移除无效查询参数，允许用户开始新运行。
- 检查点已过期：不加载内容，清理目录并提示结果已过期。

## 清理与隐私

- 默认 TTL 为 24 小时，从最后一次成功检查点写入时间计算。
- 页面加载和新运行开始时执行轻量过期清理。
- 清理只删除检查点根目录下、目录名符合 SHA-256 格式且已经过期的目录。
- 当前正在写入且更新时间未过期的运行不得删除。
- “清空当前数据”删除当前令牌对应检查点、相关 `st.session_state` 键和 `st.query_params["run"]`。
- 恢复令牌相当于短期访问凭据；页面提示用户不要分享带 `run` 参数的网址。
- 恢复令牌使用 `secrets.token_urlsafe(32)` 生成，目录只保存令牌哈希。
- 检查点根目录保持在 `.cache/`，继续由 Git 忽略，不进入 `outputs/` 和正式发布文件。
- 本地磁盘检查点可跨页面刷新和同一实例的进程重启恢复，但 Streamlit Cloud 容器重新部署或实例磁盘重置后不保证保留。

## 兼容性

- `checkpoint_callback` 默认为 `None`，其他调用者无需修改。
- `AutoWorkflowResult` 的最终字段和完整 ZIP 协议不变。
- 现有活动进度回调和完成进度回调继续工作。
- 不改变查询缓存数据库和缓存键。
- Streamlit 最低版本从 1.36 提升至 1.43，以使用不会触发重跑的下载按钮。

## 测试设计

### 检查点存储测试

- DataFrame JSON gzip 往返后列名、行数、数值、文本和缺失值一致；
- 多表、多图和模块工作簿清单可恢复；
- 临时文件未提交时不会被当作完整检查点；
- `os.replace()` 后清单成为唯一提交标记；
- 损坏 JSON、损坏 gzip、缺失表文件返回明确错误而非未处理异常；
- 无效令牌和路径越界输入被拒绝；
- 24 小时边界前保留、边界后清理；
- 显式删除只影响指定运行。

### 工作流测试

- 选定模块按依赖顺序发送检查点；
- 成功、失败和跳过模块都进入 `finished_steps`，具体终态保留在 `step_status`；
- 每个模块最多发送一个终态检查点；
- 回调为 `None` 时现有输出完全一致；
- 回调自身抛出检查点写入异常时，页面层降级逻辑不会中止查询；
- 最终检查点包含完整 `step_status`、`warnings` 和公开结果表。

### 页面测试

- 页面创建和读取 `run` 查询参数；
- 每个完成模块使用唯一稳定的下载按钮键；
- 模块下载按钮包含 `on_click="ignore"`；
- 新输入不会复用旧检查点；
- 刷新恢复后显示恢复提示、结果总览和模块下载；
- `running` 检查点恢复后显示“上次运行未正常结束”；
- “清空当前数据”删除会话状态、查询参数和磁盘检查点；
- 最终 ZIP 失败时仍显示模块工作簿和部分 ZIP；
- 检查点写入失败时显示降级警告，查询结果仍保存在会话。

### 验证

- 运行检查点、工作流、进度和上传状态的定向单元测试；
- 运行 `python -m unittest discover -s tests -v`；
- 运行 `python -m compileall app.py pages src`；
- 使用不联网的合成长任务，模拟模块成功、失败、跳过、刷新恢复和最终 ZIP 失败；
- 使用少量缓存化合物进行 Streamlit 冒烟测试，确认运行中模块下载不会触发重跑。

## 验收标准

1. 任一模块达到终态后，用户无需等待后续模块即可看到状态和下载该模块工作簿。
2. 后续模块失败或最终 ZIP 失败时，此前模块结果仍可查看和下载。
3. 页面刷新后，24 小时内可通过当前恢复网址重新加载检查点结果。
4. 下载模块结果不会触发 Streamlit 重跑或中断当前任务。
5. 新输入不会与旧检查点混合。
6. 检查点失败只影响跨刷新恢复，不影响主要查询。
7. 完整运行成功时，现有完整结果表、图表和 ZIP 协议保持不变。
8. 过期、损坏或恶意路径输入不会读取检查点根目录之外的文件。
